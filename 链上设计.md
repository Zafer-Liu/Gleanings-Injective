# 《拾遗 · Gleanings》链上设计文档

> 面向 Injective EVM 的完整链上方案：资产模型、合约、铸造与防作弊、元数据、
> 版税、网络、前后端集成、部署、安全、测试，以及对 Injective 赛道评审标准的映射。
>
> 配套合约：[`contracts/Gleanings.sol`](contracts/Gleanings.sol)（OpenZeppelin v5，已通过 solc 0.8.28 编译，字节码约 14.3KB，0 警告）
> 版本 v1.0 ｜ 更新：2026-07-23

---

## 1. 设计目标与原则

1. **玩家真正"拥有"文化**：沿途信物、通关徽章都是玩家钱包里可验证、可转让、可交易的 NFT。
2. **只把该上链的上链**：藏品上链；剧情、选择、进度等游戏逻辑留在链下。省 Gas、逻辑清晰、开发快。
3. **不可作弊**：信物必须"确实玩过那一幕"才能铸造；徽章必须"确实集齐信物"才能领取。
4. **可复用**：合约以"章节（series）+ 幕（piece）"抽象，新增一件传统事物只需 `defineSeries`，合约不改。
5. **接商业模式**：内置 ERC-2981 版税，二级交易可为品牌方/团队分成。

---

## 2. 上链 / 链下分工

| 内容 | 位置 | 说明 |
|---|---|---|
| 信物 NFT、徽章 NFT | **链上（Injective EVM）** | 唯一、可拥有、可转让、可交易 |
| 收藏馆/图鉴展示 | 链上读取 + 前端渲染 | 通过 `collectionOf(address)` / `tokenOfOwnerByIndex` 读取 |
| 剧情、对白、选择分支、轻互动 | 链下（前端） | 数据驱动 JSON，不上链 |
| 玩家进度、存档 | 链下 | localStorage / 后端 |
| 幕通关的"授权凭证" | 链下签发 → 链上校验 | 后端 EIP-712 voucher（见 §5） |

> 向评委的一句话解释：**只有"藏品"上链，因为玩家真正拥有了他走过、读懂的这份文化；其余无需上链。**

## 3. 资产模型

### 3.1 两类资产

- **信物 Relic**：每通关一幕获得一枚，记录章节、幕、稀有度、玩家选择路径与 AIGC 图。
- **徽章 Badge**：集齐某章全部信物后铸造，代表"读懂了这件传统事物"。

### 3.2 为什么用 ERC-721（而非 ERC-1155）

| 维度 | ERC-721（本项目采用） | ERC-1155（备选） |
|---|---|---|
| 唯一性 | 每枚 token 唯一，可记录玩家个人选择路径与个性化图 | 同类信物同质化，个性弱 |
| 叙事契合 | "你拥有你自己的那一段旅程" | "你拥有一张通用凭证" |
| 二级交易/稀有度 | 天然适配唯一藏品交易 | 更适合大批量同质道具 |
| Gas | Injective EVM 近乎零 Gas，唯一性成本不敏感 | 批量更省，但本项目非批量场景 |

**结论**：主打唯一收藏与个性化，采用 **ERC-721**。若未来某章需要大批量同质道具，可另部署 1155 补充。

### 3.3 数据结构（链上）

```solidity
enum Kind { Relic, Badge }
struct Piece {
    uint256 seriesId;  // 章节：1 = 福建老酒
    uint256 pieceId;   // 幕/信物编号；徽章为 0
    Kind    kind;
    uint8   rarity;    // 0 普通 … 4 传世
    bytes32 pathHash;  // 玩家选择路径哈希（个性化/防篡改）
}
mapping(uint256 => Piece) public pieces; // tokenId => 属性
```

## 4. 合约架构

**单合约 `Gleanings`**（继承 OpenZeppelin v5）：

```
Gleanings
 ├─ ERC721Enumerable   // 可枚举，图鉴/收藏馆读取
 ├─ ERC721URIStorage   // 每枚 token 唯一 tokenURI（个性化 metadata）
 ├─ ERC2981            // 二级交易版税（商业模式）
 ├─ Ownable            // 管理：定义章节、设置签名者/版税
 └─ EIP712             // 信物 voucher 的类型化签名校验
```

**核心函数一览**

| 函数 | 权限 | 作用 |
|---|---|---|
| `defineSeries(seriesId, name, requiredPieceIds[])` | owner | 定义一个章节及集齐徽章所需信物 |
| `mintRelic(voucher, sig)` | 玩家 | 凭后端 EIP-712 voucher 铸造信物 |
| `claimBadge(seriesId, relicTokenIds[], badgeUri)` | 玩家 | 集齐信物后可信地铸造徽章 |
| `collectionOf(address)` | view | 返回某地址全部 tokenId（图鉴用） |
| `setVoucherSigner / setDefaultRoyalty` | owner | 运维 |

## 5. 铸造流程与防作弊（核心）

### 5.1 信物：后端签发的 EIP-712 voucher（防止凭空铸造）

游戏逻辑在链下，若铸造开放则藏品毫无意义。采用**签名凭证（voucher）**模式：

```
玩家通关"冬酿"一幕
   │
   ▼
前端 → 后端：请求 voucher（附玩家地址、章节、幕、本次选择）
   │        后端校验"确实通关" → 用签名私钥对 RelicVoucher 做 EIP-712 签名
   ▼
后端 → 前端：返回 { voucher, signature }
   │
   ▼
前端 → 合约 mintRelic(voucher, sig)
   │        合约校验：voucher.player == msg.sender
   │                 章节存在、该幕信物未铸过
   │                 EIP-712 摘要未用过（防重放）
   │                 摘要签名者 == voucherSigner
   ▼
铸造信物 NFT 到玩家钱包，标记 voucher 已用、该幕已铸
```

**voucher 结构与 typehash**（合约与后端必须逐字一致）：

```solidity
struct RelicVoucher {
    address player; uint256 seriesId; uint256 pieceId;
    uint8 rarity; bytes32 pathHash; string uri; uint256 nonce;
}
// keccak256("RelicVoucher(address player,uint256 seriesId,uint256 pieceId,uint8 rarity,bytes32 pathHash,string uri,uint256 nonce)")
```

- **防重放**：`usedVouchers[digest]` 记录已用摘要；`nonce` 保证同一玩家同一幕不同批次的 voucher 摘要不同。
- **一幕一枚**：`hasRelic[player][series][piece]` 防止同一幕重复铸造。

### 5.2 徽章：链上可信校验（无需签名）

徽章无需后端授权——**能不能领，链上就能判定**：调用者把自己持有的信物 tokenId 传入，合约逐一校验 `ownerOf == msg.sender`、属于本章、是必需信物、且覆盖全部不同 pieceId，然后铸造徽章。这是一个漂亮的"可信、无需信任后端"的展示点。

> 设计取舍：徽章资格按"当前持有"判定。若玩家把信物卖了，则暂时不满足集齐——符合"拥有即读懂"的语义。

### 5.3 重入与检查-生效-交互

`_safeMint` 会回调接收方 `onERC721Received`。关键状态（`usedVouchers`、`hasBadge`）在 `_safeMint` **之前**置位，重入会被 `voucher used / badge claimed` 拦截，安全。

## 6. 元数据设计

### 6.1 存储方式

- **图片（像素信物/徽章）+ metadata JSON** 上传 **IPFS**（如 web3.storage / Pinata），`tokenURI` 存 `ipfs://…`。
- MVP 也可先用后端静态托管 URL，赛后迁 IPFS。

### 6.2 metadata JSON 示例（信物）

```json
{
  "name": "冬酿曲印 · 福建老酒",
  "description": "霜降下曲，冬至封坛。你亲手将古田红曲拌入糯米，看它在闽地的冬天里慢慢苏醒——这一印，记下了这坛酒为何非冬不酿。",
  "image": "ipfs://bafy…/dongniang.png",
  "attributes": [
    { "trait_type": "章节", "value": "福建老酒" },
    { "trait_type": "幕",   "value": "冬酿" },
    { "trait_type": "类型", "value": "信物" },
    { "trait_type": "稀有度", "value": "稀有" },
    { "trait_type": "选择路径", "value": "问·红曲是什么" }
  ]
}
```

- `description` 承载**背景故事**（玩家读懂的那段文化即刻成为藏品的一部分）。
- `选择路径` 由 `pathHash` 对应的可读路径，体现个性化。

## 7. 稀有度与个性化

- **稀有度**（0–4）由玩家的选择路径决定：走到隐藏 lore / 情感高点的组合触发更高稀有度，写入 `Piece.rarity` 并进 metadata。
- **个性化**：`pathHash = keccak256(选择序列)` 上链，作为"这枚信物属于你这一遍旅程"的防篡改凭据；同时可据路径选择不同 AIGC 图变体。
- 稀有度分布由后端签发 voucher 时决定，链上只忠实记录。

---

## 8. 版税与二级交易

- 合约实现 **ERC-2981**，构造时设 `royaltyReceiver` 与 `royaltyBps`（如 `500` = 5%）。
- 对接设计文档中的商业模式：**二级交易分成给品牌方（如福建老酒）/ 团队**。
- MVP 不必自建市场，藏品可在支持 Injective EVM 的 NFT 市场流转；如需现场演示交易，可加一个最简 `list/buy` 市场合约（**冲刺项，非必需**）。

## 10. 前后端集成

### 10.1 后端：签发 voucher（viem）

```ts
import { privateKeyToAccount } from 'viem/accounts'
import { keccak256, encodePacked } from 'viem'

const account = privateKeyToAccount(process.env.SIGNER_KEY as `0x${string}`) // 切勿入库/入仓

const domain = {
  name: 'Gleanings', version: '1',
  chainId, verifyingContract, // verifyingContract = 合约地址
} as const

const types = {
  RelicVoucher: [
    { name: 'player',   type: 'address' },
    { name: 'seriesId', type: 'uint256' },
    { name: 'pieceId',  type: 'uint256' },
    { name: 'rarity',   type: 'uint8'   },
    { name: 'pathHash', type: 'bytes32' },
    { name: 'uri',      type: 'string'  },
    { name: 'nonce',    type: 'uint256' },
  ],
} as const

// 玩家通关"冬酿"后：
const pathHash = keccak256(encodePacked(['string'], [choicesSerialized]))
const voucher = {
  player, seriesId: 1n, pieceId: 1n, rarity: 2,
  pathHash, uri: 'ipfs://…/dongniang.json', nonce: BigInt(Date.now()),
}
const signature = await account.signTypedData({
  domain, types, primaryType: 'RelicVoucher', message: voucher,
})
// 返回 { voucher, signature } 给前端
```

> 注意：EIP-712 会自动对 `string uri` 做 keccak256——与合约里 `keccak256(bytes(v.uri))` 一致；顺序必须与 typehash 完全相同。

### 10.2 前端：铸造信物（viem/wagmi）

```ts
import { writeContract } from '@wagmi/core'

await writeContract(config, {
  address: GLEANINGS_ADDRESS,
  abi: gleaningsAbi,
  functionName: 'mintRelic',
  args: [voucher, signature],
})
```

### 10.3 前端：领取徽章

```ts
// 读取玩家在本章持有的信物 tokenId（由后端/事件索引，或遍历 collectionOf 过滤 seriesId）
await writeContract(config, {
  address: GLEANINGS_ADDRESS, abi: gleaningsAbi,
  functionName: 'claimBadge',
  args: [1n, relicTokenIds, 'ipfs://…/fujian-badge.json'],
})
```

### 10.4 前端：读取收藏馆/图鉴

```ts
const ids = await readContract(config, {
  address: GLEANINGS_ADDRESS, abi: gleaningsAbi,
  functionName: 'collectionOf', args: [player],
})
// 对每个 id 读 pieces(id) + tokenURI(id) → 渲染信物墙/徽章墙
```

## 11. 部署与初始化

### 11.1 工具

- **Hardhat** 或 **Foundry**（Injective EVM 兼容标准 EVM 工具链）。
- 依赖：`@openzeppelin/contracts@5`。

### 11.2 部署脚本（Hardhat 示例）

```js
const signer = process.env.VOUCHER_SIGNER      // 后端签名者地址
const royaltyReceiver = process.env.ROYALTY_TO // 版税收款
const g = await ethers.deployContract('Gleanings', [signer, royaltyReceiver, 500]) // 5% 版税
await g.waitForDeployment()

// 定义第一章：福建老酒，集齐信物 = [1 冬酿曲印, 2 青花酒盏]
await (await g.defineSeries(1, '福建老酒', [1, 2])).wait()

console.log('Gleanings:', await g.getAddress())
```

### 11.3 第一章 piece 约定

| pieceId | 名称 | 来源幕 |
|---|---|---|
| 1 | 冬酿曲印 | 幕二《冬酿》 |
| 2 | 青花酒盏 | 幕三《月子酒》 |
| 0（徽章） | 福建老酒徽章 | 幕四《归》（集齐 1+2） |

## 12. 安全清单

- [ ] **签名私钥**只放后端环境变量 / KMS，**绝不入仓、不入前端**（`.gitignore` 已含 `.env`）。
- [ ] voucher 含 `nonce` 且合约记录 `usedVouchers`，防重放。
- [ ] `mintRelic` 校验 `voucher.player == msg.sender`，防他人代铸/抢铸。
- [ ] 关键状态在 `_safeMint` 前置位，防重入。
- [ ] `defineSeries` 仅 owner；上线后章节定义谨慎（当前设计每章只定义一次）。
- [ ] 版税 bps 合理（≤10%），`royaltyReceiver` 非零。
- [ ] 测试网先行；主网部署前做一次外部检视。

## 13. 测试计划

| 用例 | 期望 |
|---|---|
| 合法 voucher 铸信物 | 成功，`hasRelic` 置位，事件正确 |
| 重复用同一 voucher | revert `voucher used` |
| 他人用你的 voucher | revert `not your voucher` |
| 伪造签名 | revert `bad signature` |
| 同一幕铸两次 | revert `relic already minted` |
| 未集齐领徽章 | revert（数量/持有校验失败） |
| 集齐后领徽章 | 成功，`hasBadge` 置位 |
| 传入重复 pieceId 凑数 | revert `duplicate piece` |
| 非本人持有的信物凑数 | revert `not owner` |
| `collectionOf` | 返回全部持有 tokenId |
| `supportsInterface` | 721/Enumerable/URIStorage/2981 均 true |

建议用 Foundry（`forge test`）或 Hardhat + viem 编写。

## 14. 对 Injective 赛道评审标准的映射

| 评审标准 | 本设计如何得分 |
|---|---|
| 是否集成 Injective 区块链 | 信物+徽章真部署在 Injective EVM 测试网，可读可交易 |
| 展示与项目完整性 | 完整闭环：玩→签发→铸造→集齐→徽章→图鉴，合约已编译可部署 |
| AI 集成创新性 | AI 生成个性化信物图与叙事，路径哈希上链绑定玩家旅程 |
| 应用价值 | 传统文化出海 + 玩家真实拥有文化藏品；ERC-2981 支撑品牌分成 |
| 产品体验 | 仅关键时刻上链、近零 Gas，体验顺滑；收藏馆可视化 |

## 15. MVP 边界与取舍

- **做**：`Gleanings` 合约（信物 voucher 铸造 + 徽章可信领取 + 版税 + 枚举）；测试网部署；第一章 2 信物 + 1 徽章；前端连钱包 + 铸造 + 图鉴读取。
- **冲刺项（有余力再做）**：自建最简市场合约演示二级交易；IPFS 全量迁移；事件索引器/子图。
- **不做**：把游戏逻辑上链；复杂经济系统；跨链。
